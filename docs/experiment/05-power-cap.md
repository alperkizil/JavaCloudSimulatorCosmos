# 5. Power Ceiling — Constraints, Enforcement, and the Admission Disaster

## 5.1 What "power ceiling" means

A real datacenter has a **maximum instantaneous draw** it can support
— dictated by the electrical feed, UPS capacity, or contractual demand
charges. Going above the cap, even briefly, risks breakers tripping
and penalties. A scheduler ignorant of this cap can choose an
assignment whose average behaviour looks fine but whose **peak** blows
the ceiling.

We measure peak aggregate datacenter power with an event-driven
sweep-line (see [`03-objectives.md`](03-objectives.md) §3.5):

$$
P_{\text{peak}}(a) \;=\; \max_{t\in[0,\,\text{makespan}(a)]}\; \sum_{v\in V_{\text{active}}(t)} P_v.
$$

A solution is **feasible** if $P_{\text{peak}}(a) \le P_{\text{cap}}$
and **infeasible** otherwise. The *amount* of infeasibility is

$$
\mathrm{violation}(a) \;=\; \max\bigl(0,\; P_{\text{peak}}(a) - P_{\text{cap}}\bigr).
$$

## 5.2 How the three cap levels were chosen

We first ran the uncapped experiment, collected the peak power of
every solution on the Pareto fronts across all 10 seeds × 3 scenarios
× 7 algorithms, and built the empirical CDF of those peaks. The three
cap levels are calibrated percentiles of that CDF:

| Cap level | Tightness | Feasible fraction of uncapped solutions |
|-----------|-----------|-----------------------------------------|
| 220 kW    | loose     | ≈ 78 %                                  |
| 190 kW    | moderate  | ≈ 48 % — near the global median peak   |
| 120 kW    | tight     | ≈ 21 % — stress regime, only energy-optimizers survive |

Source: `FinalExperiment/PowerCeilingWaitingTimeExperimentRunner.java:171-182`.

The paper compares 190 kW (moderate) and 120 kW (tight) against
uncapped. The 220 kW tier is computed but left as diagnostic output
because it barely bites.

## 5.3 How constrained MOEAs know the cap

**Short answer**: it is *neither a penalty function nor a third
objective*. It is a **constraint value**, and the MOEAs combine it
with the two objectives using **Deb's constrained-domination** rules.

### 5.3.1 The constraint hook

Each solution evaluated under a cap carries:

- two **objectives** $(\mathrm{WT}(a),\; \mathrm{E}(a))$ — unchanged from
  the uncapped problem;
- one **constraint value** $c(a) = \mathrm{violation}(a) \in [0, \infty)$.

Code (constraint value is in watts):

```java
// moea/PowerCeilingSchedulingProblem.java:82-89
meter.computePowerProfileOnly(schedulingSolution, tasks, vms);
double violation = Math.max(0.0, meter.getLastPeakPower() - powerCapWatts);
solution.setConstraint(0, violation);
```

The problem declares `numberOfConstraints = 1` in its constructor
(line 50), which flips NSGA-II / SPEA-II / AMOSA into
constraint-aware mode.

### 5.3.2 Deb's constrained-domination rule

Let solutions $a$ and $b$ have objective vectors $\mathbf{f}(a), \mathbf{f}(b)$
and violation magnitudes $c(a), c(b)$. Then $a$ **c-dominates** $b$
iff any of:

1. $c(a) = 0 \wedge c(b) > 0$ — feasible beats infeasible.
2. $c(a) > 0 \wedge c(b) > 0 \wedge c(a) < c(b)$ — among infeasibles,
   smaller violation wins.
3. $c(a) = 0 \wedge c(b) = 0 \wedge \mathbf{f}(a) \prec \mathbf{f}(b)$ —
   among feasibles, standard Pareto.

$$
\boxed{\;a \prec_c b \;\;\Longleftrightarrow\;\;
\begin{cases}
c(a)=0,\; c(b)>0, \\
\text{or } c(a)<c(b),\; c(a)>0, \\
\text{or } c(a)=c(b)=0 \text{ and } \mathbf{f}(a)\prec\mathbf{f}(b).
\end{cases}\;}
$$

Why this works better than a penalty: a penalty term
$\mathrm{E}(a) + \lambda \cdot c(a)$ requires choosing $\lambda$. Too
small and the optimiser ignores the cap; too large and feasibility
crushes objective quality. Deb's rule has no magic constant — it says
"feasible always wins", which is what the operator of the datacenter
actually wants.

Why this works better than a third objective: if you added $c$ as a
third minimisation objective, the algorithm would spend budget
pushing $c$ from (say) 5 W to 3 W even though any $c \le 0$ is already
"good enough". Treating it as a constraint stops rewarding effort once
feasibility is achieved.

### 5.3.3 The GA / SA dominance-archive cap variants

For the home-grown GA and SA dominance-archive families, the same
feasible-beats-infeasible logic is applied at the **archive** level
(`metaheuristic/ConstrainedNonDominatedArchive.java`):

```java
// ConstrainedNonDominatedArchive.java:9-15
//   feasible ≻ infeasible
//   both infeasible → smaller violation dominates
//   both feasible   → standard Pareto on raw objectives
```

Every candidate generated by the GA / SA search is offered to the
archive along with its violation. The archive stores only
non-c-dominated members, so the final front returned to the simulator
is guaranteed feasible whenever at least one feasible point was found
during the run.

## 5.4 Runtime admission control — the disaster

The `WorkloadAware_Admission_PC_{cap}` baseline is a **decorator**: it
wraps the standard `WorkloadAware` heuristic and applies the cap at
the *moment of placement*, one task at a time, with no look-ahead.

### 5.4.1 What the code does

Source: `PlacementStrategy/task/PowerCeilingAdmissionTaskAssignmentStrategy.java`

```java
// lines 191-192
if (baselineWatts + incrementalWatts > powerCapWatts + 1e-6) {
    // defer this task — pick a later slot where the cap still fits
}

// lines 235-244 — after defer, re-check the future slot
if (baselineWatts + cumulative + incrementalWatts > powerCapWatts + 1e-6) {
    return false;       // still doesn't fit, keep deferring
}
return baselineWatts + cumulative + incrementalWatts <= powerCapWatts + 1e-6;
```

The strategy:

1. For each incoming task, estimate `baselineWatts` (current DC draw)
   and `incrementalWatts` (what the task would add).
2. If the sum exceeds `P_cap`, refuse this assignment and push the
   task to a later time slot.
3. Recheck at the later slot; if still above cap, keep deferring.

### 5.4.2 Why it blows up

Deferring is *not* free. Every deferred task still has to run
eventually, and the scheduler has no plan for when. Because:

- there is **no look-ahead**: the strategy cannot see that admitting
  task A *now* is fine but would crowd out task B *in 5 seconds*;
- there is **no re-planning**: deferred tasks are revisited one at a
  time, in arrival order, with the same greedy check;
- there is **no rebalancing**: a better VM might exist for the
  deferred task but the strategy only ever tries the one
  `WorkloadAware` originally picked.

The result is a **cascading queue**: tasks refused at $t_0$ retry at
$t_1$ and often get refused again, while new arrivals also get refused.
Waiting time stacks up linearly with every refusal. Measured blow-up:
waiting time grows ≈ 22–50× compared with uncapped `SPEA-II`, while
energy barely improves.

See `final_experiment_results/admission_disaster.png` for the
log-scale visual of the blow-up, and `pivot_TimeMs.csv` for the
wall-clock that makes the failure mode unambiguous.

### 5.4.3 Why the optimiser-based arms escape this

NSGA-II / SPEA-II / AMOSA with the constrained problem plan the
**entire batch of 500 tasks together**. They search for one
assignment in which $P_{\text{peak}}(a) \le P_{\text{cap}}$ holds
globally, using the constraint value to steer the population toward
feasible regions. Because the whole plan is feasible, no task ever
has to be refused at runtime and no queue cascade can form.

Put differently: the admission-control baseline commits locally and
then *hopes* the global constraint is satisfied; the constrained
MOEAs commit globally and then *guarantee* the constraint was
satisfied. That is the whole headline of the power-ceiling half of
the paper.
